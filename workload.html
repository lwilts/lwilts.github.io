Workloads, Applications, and Services

What’s in a name?

In the fast-moving and often confusing world of technology and IT, we’re used to words having multiple meanings and interpretations. Every organisation has varying terminology, and many words we use daily have very different meanings in different contexts.

Three prime examples of this are “workload”, “application” and “service”, which come up a lot when we discuss cloud migration.

Let’s look at some of the available definitions of these.

Workload

“The amount of work to be done, especially by a particular person or machine in a period of time.”
— Cambridge Dictionary

This generic definition implies a rate of work rather than a single complete task, which differs from what we might think of as a computing workload.

“Any program or application that runs on any computer.”
— techtarget.com

This might be a finite process, such as a batch processing job, or a long-running one, such as an operating system.

In the cloud era, the definition evolves. According to AWS:

“A collection of resources and code that delivers business value, such as a customer-facing application or a backend process.”

This is broader, but it makes sense in a cloud environment where compute, networking, and storage are all virtualised. A database might be a workload in itself, but when tightly coupled with a frontend interface, we could call this one workload.

Application

“A computer program that is designed for a particular purpose.”
— Cambridge Dictionary

This sounds similar to our computing definition of a workload. The key differentiator seems to be the notion of “purpose”.

We often prepend the word application (or app) with a purpose: email, banking, dating, backup.

This definition scales across different situations:

  • A mobile app
  • A backend system (e.g., inventory or CMS)

A modern application might just be one component—like a frontend—that connects to other applications, often via APIs.

Distinguishing Applications from Workloads

A good way to distinguish the two:

  • A developer writes an application (code stored in GitHub or published to an app store).
  • It becomes a workload when it’s run somewhere—a VM, laptop, phone, or Kubernetes cluster.

Service

If you’re a Linux sysadmin, a service is:

“An application (or set of applications) that runs in the background waiting to be used, or carrying out essential tasks.”
— linux.com

We know how to create services using systemd to keep programs running.

At the other end of the spectrum, when discussing user-facing systems:

“Something that helps [the user] to do something – like learn to drive, buy a house, or become a childminder.”
— Government Digital Service

These human-facing services require considerations for user experience and UI design.

In cloud computing, we also hear terms like:

  • Infrastructure as a Service (IaaS)
  • Software as a Service (SaaS)

Here, the user gets an interface without needing to run or configure the underlying systems.

“A mechanism to enable access to one or more capabilities.”
— OASIS

This leans toward the access mechanism itself being the "service"—arguably accurate in systemd and Kubernetes contexts.

But in the government service sense, a service tends to include:

  • The user journey
  • The apps used along the way
  • The workloads that support those apps

Microservices

Now, let’s throw a spanner in the works and talk about microservices.

Microservices – or microservice architecture – is an architectural style that structures an application as a collection of services that are:
  • Highly maintainable and testable
  • Loosely coupled
  • Independently deployable
  • Organised around business capabilities
  • Owned by a small team
— microservices.io

An application built with microservices is composed of several underlying services, each serving a distinct purpose via an API.

Each of these microservices could be:

  • Considered an application in its own right
  • Referred to as a workload when deployed (e.g. containers, DB, config)

But is a microservice a “service” in the user experience or government sense? Absolutely not.

Workload Migration

Workloads are what we at AL prefer to discuss when carrying out cloud migrations — after all, if you aren’t running your applications somewhere, they aren’t providing much value.

Typically, a workload is the minimum viable unit for migration because it includes:

  • The application
  • Supporting resources (e.g., storage, networking)

Workloads are what underpin the business value of a system.

Sometimes, several related workloads are grouped into a migration group based on:

  • Connectivity
  • Latency
  • Security
  • Shared deployment patterns

Concluding thoughts

In many ways, workloads, applications, and services are just different sides of the same (geometrically improbable) coin.

The subtle differences:

  • A workload is an instantiation of an application.
  • A service is how that capability is consumed, by a user or another system.

There’s a fractal-like pattern to these terms:

  • Top level: a user interacts with a service → via an application → backed by a workload.
  • Zoom in: the application may include other applications, services, and workloads.
  • Zoom in further: code-level functions or modules could be seen as workloads by a developer.

Essentially — it’s all relative!

Final Definitions

  • Workload: A tightly coupled group of resources which run and support an application or capability.
  • Application: A piece of software which fulfils a specific purpose.
  • Service: The means of facilitating interactions with a workload in order to utilise its capabilities and fulfil a job to be done.

Ultimately, what matters is that what you create provides business value. It’s very difficult to use terminology in a completely consistent way, but that’s OK — as long as users get value, and teams can collaborate with shared meaning to generate that value.
